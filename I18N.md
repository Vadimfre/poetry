# Content localization (API + DB)

## Locales

Supported: `be` (default), `ru`, `en`.

## Request locale

Every HTTP request resolves locale in this order:

1. Header `X-Locale: be|ru|en`
2. Query `?lang=be|ru|en`
3. Header `Accept-Language`
4. Default `be`

Frontend sends `X-Locale` and `Accept-Language` from the language cookie (`poetry_locale`).

## Database shape

Translatable entities store JSON in column `i18n`:

```json
{
  "be": { "title": "...", "content": "..." },
  "ru": { "title": "...", "content": "..." },
  "en": { "title": "...", "content": "..." }
}
```

Entities: `Poem`, `Author`, `Category`, `Holiday`, `SeasonSlide`, `Quiz`, `Question`, `Item`, `Zone`.

Scalar columns (`title`, `content`, `bio`, …) remain the primary Belarusian copy for search and admin; API responses return localized scalars and omit raw `i18n`.

## Fallback

If a field is missing for the requested locale: `locale` → `be` → legacy scalar column.

## Admin

Creating content via admin sets `i18n.be` from submitted fields. To add RU/EN, update `i18n` in the database (or extend admin forms).

Example SQL for a poem:

```sql
UPDATE "Poem"
SET "i18n" = jsonb_set(
  "i18n",
  '{ru}',
  jsonb_build_object('title', 'Русский заголовок', 'content', '...', 'description', '')
)
WHERE id = 1;
```
